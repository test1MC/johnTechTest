Elasticity Model Inputs and Calculations

 

Elasticity Overview

 

Overview: Price Elasticity of Demand

Elasticity forms the foundation of PriceTek’s precision pricing, measuring deposit demand as a function of your bank’s competitive pricing position, measured against internal and competitor alternative products.

Generally, customer price elasticity is non-linear, and inflection points exist above or below which a more substantial response can be measured.

 

Inputs: Flow of Funds

Customer response is the primary dependent variable in PriceTek's elasticity models. This is measured using our proprietary flow of funds methodology, calculating customer level balance movements across several core balance flows. Individual models are built for each flow.

 

Elasticity Model Data Sources

Elasticity curves are built for each portfolio segment and flow combination using the bank’s balance flows compared to a relative price position. The balance flows are calculated using only the bank’s performance. The relative rate captures the bank’s offered rate relative to what competitors are offering for a similar portfolio segment.

Note: Industry data is captured only in the relative rate for acquisition and percent roll.

 

 

Model Design

Relative rates and balance flows are combined to create the final model input data set that is used to calculate regressions.

  • The final model input data set consists of the historical sets of independent and dependent variables for each model, with each month in the history considered as a data point
  • Novantas validates the input data set prior to running the regressions
    • Data points are evaluated to ensure they have populated values for the independent and dependent variables. If any component is missing, the data point will be excluded.
    • The data is examined for outliers and data anomalies. Points that do not reflect customer driven behavior, such as product migrations or bank acquisitions, are excluded from the models.
  • The final data set is limited by certain assumptions
    • Based on data availability, minimum account thresholds are set to create each data point. A limit applies to acquisition data points, with a separate limit applying to the other modeled flows which are calculated as a percentage.
  • After the model input data is refined, the regression software calculates an elasticity model for each portfolio segment and balance flow combination
    • Pricing cells having an inadequate number of data points, or a negative correlation between relative rate and acquisition, will be given historical average models based on the average acquisition across the data points.

Our model variables are calculated differently across balance flows.

Y-Axis: The Dependent Variable

Flow Level Models

 

Understanding Acquisition

Y-Axis: Modeled Acquisition

Modeled acquisition combines the acquisition and switch in augmentation balance flows calculated for each month across pricing cells using your institution's (those with physical branches) month-end account data.

Modeled acquisition is calculated at a “per-branch” level (again, for those institutions with physical branches) in order to normalize for the number of branches in a given market across time. In addition, modeling at this level allows for a commensurable comparison of acquisition across markets.

X-Axis: Relative Rate Calculation

The Acquisition relative rate measures how competitively your institution is priced by comparing the weighted rate of each pricing cell to an Acquisition Reference Rate. This acquisition reference rate is calculated using an external reference rate and an internal reference rate.

  • The external reference rate for each pricing cell is chosen from the set of reference rates available in PriceTek, based on which reference rate maximizes r2 for the model. The external reference rate consideration set can be limited by portfolio segment and is customizable. Your institution’s current settings for the consideration set can be found here.
  • For some models, the acquisition reference rate is adjusted to include the effect of internal competition offers from your own institution which may cannibalize acquisition if sufficiently attractive.
    • For example, a 140 BPS rate on a CD would drive more acquisition if the Savings is 100 BPS than if it is 150 BPS.

For each cell, the Internal Reference Rate is defined as the max rate across your institution's products considered to compete for acquisitions.
  • For example, Checking products would compete for acquisition with Savings.
  • The cross product reference rate is calculated (see right).

For modeling, we create a new final “reference rate” that considers both the external reference rate (e.g. Weighted Max) and the internal reference rate.

Note: By product, only certain competing internal products are eligible for consideration as part of the Internal Reference Rate.

Understanding Percent Roll

Y-Axis: Percent Roll

Percent Roll measures the balances that remain in the institution in a given month. Attriting balances are subtracted from starting balances to determine balances that have rolled – or stayed at the bank – despite the opportunity to leave.

Note: For term products, only the maturing component of on book balances are used since only these balances are considered to have the opportunity to leave.

Roll balance total is then converted to a percentage of starting balances, with the result being called Percent Roll.

We use Percent Roll for our Y-axis as opposed to Roll because Roll is a function of what is on book in any given month of data. We report it as a percent to normalize it month-to-month and to account for the fact that one month might have a larger amount of balances come to maturity than another.

X-Axis: Relative Rate Calculation

We have generally found that, for term products, Attrition and Roll behavior is a function of both today’s rate position relative to the market rate and the rate position at which the customer last renewed or at which was acquired.
  • To capture both factors, we use a Proxy Reference Rate that is a weighted average of both these components rather than just considering the market rate (see right).
  • We then create a new final Relative Rate that considers both the Proxy Reference Rate and the current APY as our X-Axis (see right).
  • The weighted M2 APY includes the Switch-to rate and competing internal offers. Thus, it eliminates the need for any further cross-product adjustment.

 

Understanding Percent Switch

Y-Axis: Percent Switch Flow Calculation

For an individual data point, Percent Switch is calculated by dividing the Switch Out balances by the Roll Balances.

Percent Switch measures the balances that leave the product to fund a new account or product.

 

X-Axis: Relative Rate Calculation

For Percent Switch models, we take into account the rates that influence the “opportunity cost” of staying with the original product. For that reason, we include the Switch Out and Switch In rates in the relative rate calculation.
  • The likelihood of switch is a function of the relative attractiveness of the rate of the switch-to product compared to the rate of the switch-from product.

Switch In Balance Allocation

PriceTek models generate a Switch Out prediction. To produce an accompanying Switch In prediction, we allocate the predicted Switch Out balances back into the portfolio using a simplifying assumption: these balances are rate seeking and will move to the highest rate option available.

  • For each product and term, we assume a set of other products that balances can switch into; these assumptions are determined based on customers’ actual switch behavior as seen in PriceTek, the bank’s strategy, and Novantas experience.
  • The Scenario Engine chooses the product out of the set with the highest on-sale rate as the Switch-In product and allocates all switch out balances from the particular product and term to it.
  • The Switch-In balance for any particular product is calculated by aggregating its assigned Switch-In balance from the entire portfolio

Understanding Percent Change Stay

Percent Change Stay Intro

Percent Change Stay is a largely inelastic flow that we replace with a historical average that captures seasonal effects in liquid products.

 

Example Flow Calculation

Of the balances that roll, the month over month augmentation or diminishable amount is measured by the Percent Change Stay flow.

 

Seasonality Adjustment

Rate is not the primary driver of Change to Existing for liquid products – instead, we model seasonality.

Factoring in the Number of Fridays

We also normalize change to existing by the number of pay periods in a given month for checking products.

Segmentation Basics

 

Rate Responsiveness

To identify the differences in rate responsiveness among customer segments, we create models at a pricing cell level.

Elasticity models are calibrated for each combination of line of business, pricing market, product group, tier bucket, and term bucket in the portfolio.

Additional Retention Segmentation for Term Products

For term products, we see different retention behavior based on the number of maturities, and model these accordingly. At a high level, accounts at their first maturity event tend to have a lower percent of balances roll than accounts that have matured at least one time in the past. Because of this common behavior, term product models in PriceTek will be built separately for various maturity profiles.

 

Retention Modeling Consolidation

Due to data insufficiency and high granularity of retention models, further consolidations may be made to ensure model robustness.

Click to see retention model consolidations, if any, which have been defined for your institution.

  • There are some nuances with Percent Roll and Percent Switch models because they are modeled as percentages
    • We require more underlying data for each data point due to the potential for large variability in the modeled flow. For example, if a Percent Roll data point only has a single underlying account, that customer’s roll decision will result in 100% roll. If they attrite, the data point will reflect 0% roll.
    • We are able to build more reliable models with more data and apply them at a more granular level
  • Using business and statistical rules, we have developed a segmentation scheme that consolidates cells to allow sufficient maturing balances for modeling
  • This segmentation scheme is designed to consolidate cells with similar retention behavior, and we use Percent Roll as our metric for determining the similarities
    • Percent Roll has an event in every month, providing more data points to determine behavior similarities between cells. Percent Switch is episodic, providing fewer data points for comparison.
    • Percent Roll is a measure of balances that stay with the bank and includes Switch behavior
  • The segmentation scheme developed using Percent Roll data is applied to Percent Switch.

 

 

Calibration and Configuration

 

 
Data Exclusions and Outliers

 

Data Exclusions

We limit data points used in models to meet three different goals.

Reflect Current Product Structure Exclude Known Anomalies Exclude Outliers
For examples, situations where we may have shortened the date range due to historical changes in product strategy or age For example, the exclusion of branch acquisitions, mergers, or back-end product re-classifications
  • We have established a methodology to statistically identify and remove model outliers where present
  • The methodology is based on the assumption that demand for bank deposits are correlated with a higher relative rate
  • Data points that fall far outside this pattern are anomalous and should be excluded

 

Identifying Outliers

Our approach to identifying outliers is based on the assumption that the price offered on a deposit product and demand are positively correlated.

  • First, we standardize each month’s relative rate1 and volume based on the mean and standard deviation of each modeled cell’s distribution
  • Then, we divide the rate vs. volume space into four quadrants based upon the distance from the mean (measured in standard deviations), as in the presented graphic
  • The cut-offs used in defining outlying regions (as seen in the graphic) were determined with an expectation that they will contain about 3-6% of all data points
    • These cut-offs can be adjusted and are monitored on an ongoing basis to ensure they’re appropriate
    • Further, manual adjustments to the outlier tagging can be made

    Additionally, it improves model fit on average by 6% across in our development dataset (as measured by R²)

1The relative rate calculation used in the outlier methodology is a simplified calculation and will not reflect the final relative rate of the model. SAS determines weights for some of the rates involving cross-product elasticities iteratively, and these weights are not available prior to the process submission. Consequently, we need to approximate the weights and use those approximated values in our relative rate calculations.

Ongoing Model Recalibration

 

Elasticity models are refreshed regularly (on a contractually-defined cadence) to incorporate the most recent data available and adjust the models if needed. Upon each elasticity model "Refresh":

  • The latest monthly data points are added to the model input data set
  • Regressions are recalculated with updated input data set
  • Novantas reassesses model performance and validates the new curves after the new data points are added
  • Pre-refresh curves are compared to the newly calibrated curves.
    • Data sets are examined for outliers.
    • Scenario output using the new models is compared to actual balance flows to gauge accuracy.
    • Client custom modeling assumptions and settings can be reassessed, if necessary.
  • A model performance report is run to highlight models where the error in recent months is highest.
    • If there are trends showing high error, Novantas will run analysis that examines the time range of historical data that is applied to the models and the resulting error for the most recent months of data.
    • This analysis helps inform if there are cells where the historical data should be limited in order to improve the predictability of the models.

Applying Models to Make Predictions with Scenarios

 

 
Scenario Build and Prediction Process

 

PriceTek Scenarios leverage elasticity models to analyze the expected portfolio impact of hypothetical bank or competitor rate positions.

 

Scenario Engine Calculations

A scenario is built by defining a set of hypothetical rates. These rates are then applied to elasticity models (by calculating an associated independent variable position) to generate balance flow predictions.

 

The Scenario Engine applies the model predictions and outputs across month-over-month balance glows to give a full picture of portfolio performance at the hypothetical rate.

  
 
Scenario Applications at the Pricing Cell Level

 

Scenarios allows the user to test rate changes at a product name and term level, which are then applied to the models built off aggregated cells.

  
 
Acquisition Predictions

 

Acquisition in scenarios is calculated by multiplying the acquisition per branch prediction by the current branch count for each modeled product group.

To generate the acquisition per branch prediction, the rate on the acquisition product is used – as part of modeling configuration, one specific acquisition product has been designated for each product group for this purpose.

  
 
Attrition and Switch Out Predictions

 

To generate predictions for attrition and switch out balances, the scenario rate position is fed into Percent Roll and Percent Switch models, yielding a predicted percentage of existing balances that are expected to attrite or switch, respectively. This percentage is multiplied by on book starting balances to generate predicted scenario balance flows.

  • Scenarios calculate predicted attrition and switch out for all product names and terms with maturing or month-end balances in the latest month of data.
  • Because retention models are calibrated as percentages, individual product names and terms with different rates can be applied to the models with different results

 
 
Example Scenario Calculation

 

The Scenario Engine applies predicted flows in percentages to the actual maturing balance in each month to obtain the balances flow in dollars.

 

First, the Roll Balance is calculated from the predicted Percent Roll and Attrition is calculated as the maturity balance that does not roll.

 

Then, the Roll Balance is decomposed into Switch Out and Stay by applying the predicted Percent Switch.

 

The Stay balance is found by deducting the aggregate Switch Out balance from the predicted Roll Balance.

 

Change to Existing is calculated by applying a seasonality factor to the predicted Stay balance.

 

Finally, all the flows are available in the Scenario Output.

  
 
Frequency of Events Adjustment

 

Some modeling cells may have sparse underlying data in which not every month contains enough activity to generate a modeling data point. In these cells, for Acquisition and Switch, we adjust model predictions based on frequency of events.

  
 
Extrapolation Methodology

 

We also apply Extrapolation Methodology to predictions when generating Scenario results for rate points that fall outside of the tested history.

  • The model continues at the slope of the curve on the last point that lies within the tested range.
  • When the extrapolated portion of the curve reaches a certain level beyond the final month of data, a cap is applied and the curve is straight-lined.
    • The cap can be indexed to either the min/max actual values or the min/max predicted values and set by balance flow
